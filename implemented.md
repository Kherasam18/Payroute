## 1. Project Overview
PayRoute is a distributed, intelligent payment orchestration engine designed to sit between an application and multiple payment providers (Stripe, Razorpay, PayPal, Braintree, Square). It aims to eliminate single points of failure by routing transactions dynamically based on strategy (priority, cost, success rate), and guarantees transaction integrity using Redis-backed idempotency. 

Currently, the project is in its **scaffolding phase**. The multi-module backend structure is set up using Spring Boot, PostgreSQL, Redis, and Kafka. All foundation configurations, Docker infrastructure, and base domain classes (enums, exceptions, base interfaces) have been created, but core business logic (entities, services, controllers) is yet to be implemented.

## 4. Architecture Decisions Log

| Decision | What Was Chosen | Why | Where in Code |
|----------|----------------|-----|----------------|
| **Idempotency TTL** | 24 Hours | Retains idempotency keys in Redis long enough to prevent duplicates over a 1-day retry window. | `application.yml` (`payroute.idempotency.ttl-hours`) |
| **Retry Policy** | 3 attempts, 500ms delay | Standardized short-circuit retry to absorb transient network failures before doing a full fallback. | `application.yml` (`payroute.retry.*`) |
| **Default Routing Strategy** | PRIORITY | Simpler to reason about as a default before dynamic cost/success-rate rules take over. | `application.yml` (`payroute.routing.default-strategy`) |
| **Exception Handling** | Global Controller Advice | Centralized exception-to-HTTP-status mapping for consistent API error responses. | `GlobalExceptionHandler.java` |
| **Database Schema** | UUID Primary Keys, JSONB for payloads | Ensures global uniqueness suited for distributed systems; semi-structured payload fields for flexibility across diverse payment provider responses. | Flyway Migrations |
