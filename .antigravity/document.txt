1. Problem Statement
Modern digital commerce relies on payment processing infrastructure that must be simultaneously fast, reliable, and fault-tolerant. However, organisations integrating directly with a single payment provider face a fundamental set of challenges:

Single Point of Failure: Dependency on one provider means any outage directly translates to 100% payment downtime, resulting in lost revenue and poor customer experience.
No Intelligent Routing: Businesses cannot optimise routing based on transaction cost, provider success rates, or geographic performance without a dedicated orchestration layer.
Duplicate Transaction Risk: Without a robust idempotency mechanism, network timeouts and retries can cause customers to be charged multiple times for the same order.
Manual Failover: Engineering teams must manually intervene when a provider fails, creating significant operational overhead and delayed recovery windows.
Lack of Observability: There is no centralised view of transaction health, provider performance, or failure trends across multiple payment processors.
Scalability Bottlenecks: High-volume transaction periods overwhelm single-threaded or synchronous processing pipelines, causing latency spikes and request drops.

These gaps collectively represent a significant operational and financial risk for any business processing digital payments at scale. A distributed, intelligent orchestration layer is needed to solve these problems in a unified, production-grade system.

2. Purpose
PayRoute is built to serve as a centralised payment orchestration engine — a middleware layer that sits between an application and multiple payment providers. The core purpose of this project is threefold:

2.1  Eliminate Single Points of Failure
By integrating 5+ payment providers (Stripe, Razorpay, PayPal, Braintree, Square) and implementing automatic provider fallback, PayRoute ensures that no single provider failure can bring down the entire payment system. Transactions are rerouted seamlessly without any user-facing interruption.
2.2  Enable Intelligent, Strategy-Driven Routing
PayRoute applies configurable routing strategies — priority-based, cost-optimised, and success-rate-driven — to dynamically select the most appropriate provider for every transaction. Strategy switching requires no code deployments, enabling runtime reconfiguration.
2.3  Guarantee Transaction Integrity
Through Redis-backed idempotency keys, PayRoute ensures that every payment attempt, regardless of retries or network failures, is processed exactly once. This protects both the business and the customer from the risks of duplicate charges.
2.4  Demonstrate Production-Grade Engineering
This project showcases end-to-end distributed systems engineering covering event-driven architecture with Apache Kafka, containerised deployment with Docker and Kubernetes, and full observability with Prometheus and Grafana, reflecting real-world enterprise-level practices.

6. High-Level Architecture
PayRoute follows a modular, event-driven microservices architecture composed of four primary service modules:

Module: Responsibility
api-gateway: Receives inbound payment requests, validates input, and enforces idempotency via Redis before forwarding to the orchestrator service
orchestrator: Applies the configured routing strategy to select a provider, delegates to the adapter layer, handles retries and automatic fallback on failure, and publishes Kafka lifecycle events
provider-adapter: Contains one adapter implementation per payment provider, each translating the internal PaymentRequest model into the provider-specific API contract
event-processor: Consumes Kafka events, updates transaction state in PostgreSQL, schedules retries, and feeds custom metrics to the Prometheus endpoint


All modules communicate asynchronously via Apache Kafka for non-blocking event propagation across the system. The orchestrator communicates synchronously with provider adapters to obtain real-time transaction results. Redis provides a shared idempotency and caching layer accessible to both the API gateway and the orchestrator. All services are containerised and orchestrated via Docker Compose locally and Kubernetes in deployed environments.
